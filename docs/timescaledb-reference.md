## Architecture Overview

### Data Flow

```
┌────────────────────────────────────────────────────────────┐
│ BASE TABLE (Source)                                        │
│ • timeframe_data_1min OR timeframe_data                   │
│ • Pure 1-minute OHLCV data                                 │
│ • 5 years history (~1.05 billion rows)                     │
│ • TimescaleDB hypertable with compression                  │
└────────────────────────────────────────────────────────────┘
                    ↓ derives via continuous aggregates
┌────────────────────────────────────────────────────────────┐
│ CONTINUOUS AGGREGATES (Pre-computed, < 2s queries)        │
│                                                            │
│ During Population:                      After Rename:     │
│ • timeframe_data_1min_5min       →  timeframe_data_5min   │
│ • timeframe_data_1min_15min      →  timeframe_data_15min  │
│ • timeframe_data_1min_1hour      →  timeframe_data_1hour  │
│ • timeframe_data_1min_1day       →  timeframe_data_1day   │
└────────────────────────────────────────────────────────────┘
```

### Key Principles

1. **Base table contains ONLY 1-minute data** - Never query directly for hourly/daily
2. **Use continuous aggregates** - Pre-computed views, extremely fast
3. **No timeframe column** - Each aggregate is a separate view
4. **Timestamps in IST** - Stored as TIMESTAMP (no timezone), implicitly IST
5. **Primary key (ticker_id, timestamp)** - Optimized for single-ticker queries

## Table Schemas

### Base Table: `timeframe_data_1min` → `timeframe_data`

```sql
CREATE TABLE timeframe_data (
    id INTEGER GENERATED BY DEFAULT AS IDENTITY,
    ticker_id INTEGER NOT NULL,
    timestamp TIMESTAMP NOT NULL,  -- IST timezone
    open NUMERIC(12,2) NOT NULL,
    high NUMERIC(12,2) NOT NULL,
    low NUMERIC(12,2) NOT NULL,
    close NUMERIC(12,2) NOT NULL,
    volume BIGINT,
    open_interest BIGINT,
    PRIMARY KEY (ticker_id, timestamp)
);
```

**Key Points:**
- NO `timeframe` column (clean schema)
- `timestamp` is TIMESTAMP (not TIMESTAMPTZ) - data is in IST
- Primary key optimized for (ticker_id, timestamp) queries
- TimescaleDB hypertable with compression

### Continuous Aggregates: All Timeframes

```sql
-- Structure is IDENTICAL for all aggregates (5min, 15min, 1hour, 1day)
CREATE MATERIALIZED VIEW timeframe_data_1min_5min AS
SELECT
    ticker_id,
    time_bucket('5 minutes', timestamp) AS timestamp,
    first(open, timestamp) AS open,
    max(high) AS high,
    min(low) AS low,
    last(close, timestamp) AS close,
    sum(volume) AS volume,
    sum(open_interest) AS open_interest
FROM timeframe_data_1min
GROUP BY ticker_id, time_bucket('5 minutes', timestamp);

-- Index for fast queries (< 2s requirement)
CREATE INDEX idx_timeframe_data_1min_5min_ticker_time
ON timeframe_data_1min_5min (ticker_id, timestamp DESC);
```

**Column Definitions:**
- `ticker_id` - Foreign key to tickers.id
- `timestamp` - Start of the bucket (e.g., 09:15:00 for 09:15-09:20 5-min bar)
- `open` - First price in the period
- `high` - Highest price in the period
- `low` - Lowest price in the period
- `close` - Last price in the period
- `volume` - Sum of volume in the period
- `open_interest` - Sum of open interest (usually NULL for equities)

## Query Patterns for < 2s Performance

### ✅ FAST: Query Continuous Aggregates (< 0.5s)

```sql
-- Get 5 years of hourly data for a single ticker
SELECT * FROM timeframe_data_1min_1hour
WHERE ticker_id = 123
ORDER BY timestamp;
-- Execution time: 0.1-0.5 seconds ✅
```

### ❌ SLOW: On-the-Fly Aggregation (2-5s+)

```sql
-- DON'T DO THIS! Aggregating 472,500 rows in real-time
SELECT
    time_bucket('1 hour', timestamp) AS hour,
    first(open, timestamp) AS open,
    max(high) AS high,
    min(low) AS low,
    last(close, timestamp) AS close
FROM timeframe_data
WHERE ticker_id = 123
GROUP BY hour
ORDER BY hour;
-- Execution time: 2-5+ seconds ❌
```

### Performance Rules

1. **Always use continuous aggregates for historical queries**
2. **Base table (timeframe_data) only for real-time 1-min data**
3. **Index leverages:** `(ticker_id, timestamp DESC)` index on all views
4. **Filter by ticker_id first**, then timestamp
5. **Avoid** `SELECT *` on base table without filters

## Example Queries

### Get Latest 100 Hours for a Ticker

```sql
-- Using ticker symbol
SELECT
    t.symbol,
    th.timestamp,
    th.open,
    th.high,
    th.low,
    th.close,
    th.volume
FROM timeframe_data_1min_1hour th
JOIN tickers t ON th.ticker_id = t.id
WHERE t.symbol = 'RELIANCE'
ORDER BY th.timestamp DESC
LIMIT 100;
```

### Get Full 5-Year History for Multiple Tickers

```sql
-- Get hourly data for RELIANCE, TCS, INFY
SELECT
    t.symbol,
    th.timestamp,
    th.open,
    th.high,
    th.low,
    th.close,
    th.volume
FROM timeframe_data_1min_1hour th
JOIN tickers t ON th.ticker_id = t.id
WHERE t.symbol IN ('RELIANCE', 'TCS', 'INFY')
ORDER BY t.symbol, th.timestamp;
-- Execution time: 1-2 seconds for 3 tickers × 5 years ✅
```

### Get Today's 5-Minute Data

```sql
-- Intraday 5-minute bars for today
SELECT
    t.symbol,
    tf.timestamp,
    tf.open,
    tf.high,
    tf.low,
    tf.close,
    tf.volume
FROM timeframe_data_1min_5min tf
JOIN tickers t ON tf.ticker_id = t.id
WHERE t.symbol = 'RELIANCE'
  AND tf.timestamp >= DATE_TRUNC('day', NOW())
ORDER BY tf.timestamp;
```

### Get Daily Candles for Last 365 Days

```sql
-- Daily bars for the last year
SELECT
    t.symbol,
    td.timestamp::DATE AS date,
    td.open,
    td.high,
    td.low,
    td.close,
    td.volume
FROM timeframe_data_1min_1day td
JOIN tickers t ON td.ticker_id = t.id
WHERE t.symbol = 'RELIANCE'
  AND td.timestamp >= NOW() - INTERVAL '365 days'
ORDER BY td.timestamp;
```

## Index Usage Guidelines

### Available Indexes

All continuous aggregates have this index:
```sql
CREATE INDEX idx_timeframe_data_XXX_ticker_time
ON timeframe_data_XXX (ticker_id, timestamp DESC);
```

### How PostgreSQL Uses These Indexes

1. **Single ticker queries:** Index scan on `(ticker_id, timestamp)`
2. **Date range filtering:** Index skip scan (very fast)
3. **ORDER BY timestamp DESC:** Uses index directly (no sort needed)

### Query Planning

Check if your query uses the index:
```sql
EXPLAIN ANALYZE
SELECT * FROM timeframe_data_1min_1hour
WHERE ticker_id = 123
ORDER BY timestamp DESC
LIMIT 100;
```

Look for:
- `Index Scan using idx_timeframe_data_1min_1hour_ticker_time`
- Execution time < 1ms

## Migration Notes

### Handling Both Table Naming Conventions

Your Tiphub code should support BOTH naming conventions:

```python
# Check which naming convention is active
def get_timeframe_table(conn, timeframe: str):
    """
    Auto-detect table naming convention
    Returns: table name (with or without _1min_ prefix)
    """
    cursor = conn.cursor()

    # Try final naming convention first
    final_name = f"timeframe_data_{timeframe}"
    cursor.execute("""
        SELECT EXISTS (
            SELECT FROM information_schema.tables
            WHERE table_name = %s
        )
    """, (final_name,))

    if cursor.fetchone()[0]:
        return final_name

    # Fall back to working naming convention
    working_name = f"timeframe_data_1min_{timeframe}"
    cursor.execute("""
        SELECT EXISTS (
            SELECT FROM information_schema.tables
            WHERE table_name = %s
        )
    """, (working_name,))

    if cursor.fetchone()[0]:
        return working_name

    raise Exception(f"No table found for timeframe: {timeframe}")
```

### Base Table Detection

```python
def get_base_table(conn):
    """
    Auto-detect base table name
    Returns: 'timeframe_data' or 'timeframe_data_1min'
    """
    cursor = conn.cursor()

    # Check for timeframe_data first (final name)
    cursor.execute("""
        SELECT EXISTS (
            SELECT FROM information_schema.tables
            WHERE table_name = 'timeframe_data'
        )
    """)

    if cursor.fetchone()[0]:
        return 'timeframe_data'

    # Fall back to timeframe_data_1min
    cursor.execute("""
        SELECT EXISTS (
            SELECT FROM information_schema.tables
            WHERE table_name = 'timeframe_data_1min'
        )
    """)

    if cursor.fetchone()[0]:
        return 'timeframe_data_1min'

    raise Exception("No base table found!")
```

## Troubleshooting

### Query is Slower Than Expected (> 2s)

1. **Check if using continuous aggregate:**
   ```sql
   -- Good
   SELECT * FROM timeframe_data_1min_1hour WHERE ticker_id = 123;

   -- Bad (aggregating on-the-fly)
   SELECT time_bucket('1 hour', timestamp), ...
   FROM timeframe_data WHERE ticker_id = 123 GROUP BY ...;
   ```

2. **Verify index usage:**
   ```sql
   EXPLAIN ANALYZE
   SELECT * FROM timeframe_data_1min_1hour WHERE ticker_id = 123;
   -- Should show "Index Scan using idx_..."
   ```

3. **Check aggregate refresh status:**
   ```sql
   SELECT * FROM timescaledb_information.continuous_aggregate_stats
   WHERE view_name LIKE 'timeframe_data%';
   ```

### Aggregate Not Updating

1. **Check refresh policy:**
   ```sql
   SELECT * FROM timescaledb_information.jobs
   WHERE proc_name LIKE '%continuous_aggregate%';
   ```

2. **Manual refresh (if needed):**
   ```sql
   CALL refresh_continuous_aggregate('timeframe_data_1min_1hour', NULL, NULL);
   ```
